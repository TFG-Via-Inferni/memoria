\begin{abstract}
En este capítulo se formaliza el alcance del proyecto y se 
establecen las bases técnicas para su desarrollo. El objetivo es 
traducir la visión conceptual de Via Inferni en una lista estructurada de 
requisitos funcionales, aplicando una primera iteración de la metodología Feature 
Driven Development (FDD). Asimismo, se define la arquitectura de software que soportará
dichas funcionalidades, detallando los entornos de producción, las herramientas de 
diseño de datos (preproducción) y la estrategia de pruebas necesaria para asegurar 
la calidad del software.
\end{abstract}

\section{Diseño arquitectónico}
\label{sec:arquitectura}

Para garantizar la escalabilidad y mantenibilidad del proyecto, se ha 
diseñado una arquitectura dividida en tres subsistemas claramente diferenciados: 
producción (tiempo de ejecución), preproducción (datos y herramientas) y validación (pruebas).

\subsection{Sistema de Producción (Runtime)}
El núcleo del juego sigue una arquitectura orientada a componentes desacoplados mediante 
eventos, evitando dependencias directas que rigidicen el código.

\begin{itemize}
    \item \textbf{Patrón Singleton para Managers:} Se utilizan gestores globales 
    (\texttt{GameManager}, \texttt{SoundManager}, \texttt{LevelManager}) responsables 
    de orquestar el flujo del juego, la persistencia de datos entre escenas y la 
    gestión de estados globales.
    \item \textbf{Arquitectura de Eventos:} La comunicación entre sistemas dispares 
    (por ejemplo, cuando el jugador recibe daño y el HUD debe actualizarse) se realiza 
    mediante un bus de eventos. Esto permite que la lógica de combate 
    no conozca la existencia de la interfaz de usuario, facilitando el trabajo paralelo de 
    los dos integrantes del equipo.
    \item \textbf{Composición sobre Herencia:} Las entidades del juego se construyen 
    mediante la agregación de componentes funcionales (Movimiento, Salud, Atacante) en 
    lugar de largas cadenas de herencia, aprovechando el modelo Component-Entity de Unity.
\end{itemize}

\subsection{Sistema de Preproducción (Datos)}
Dado que el juego requiere un balanceo constante de 15 tipos de enemigos,
múltiples ítems y estadísticas de personajes, se ha separado la lógica de los datos.

\begin{itemize}
    \item \textbf{ScriptableObjects:} Se utiliza este contenedor de datos nativo de
    Unity para definir las propiedades de enemigos, configuraciones de niveles y 
    atributos de ítems. Esto permite ajustar la dificultad (daño, velocidad, vida) 
    directamente desde el editor sin necesidad de modificar el código fuente ni recompilar, 
    agilizando las iteraciones de diseño.
    \item \textbf{Prefabs Variantes:} Se emplea un sistema de \textit{Prefabs} para 
    crear variaciones de enemigos o salas base, permitiendo propagar cambios generales 
    (como un cambio en la lógica de detección) a todas las instancias específicas 
    automáticamente.
\end{itemize}

\subsection{Sistema de Pruebas (QA)}
Para asegurar la estabilidad de las versiones que se fusionan a la rama 
\texttt{develop}, se ha establecido un protocolo de pruebas multinivel:

\begin{itemize}
    \item \textbf{Pruebas Unitarias (Lógica):} Verificación de algoritmos aislados que 
    no dependen del motor gráfico, como el cálculo de daño, la gestión de inventario o 
    la lógica de la máquina de estados.
    \item \textbf{Pruebas de Integración (Play Mode):} Validación de interacciones 
    complejas dentro del motor, como la generación correcta del mapa procedural 
    (asegurando que siempre existe un camino viable) o la detección física de colisiones.
    \item \textbf{Validación Funcional (DoD):} Como se mencionó en la metodología, 
    ninguna funcionalidad se da por cerrada hasta que ambos desarrolladores verifican 
    visualmente en una \textit{build} que la mecánica cumple los requisitos de la lista 
    de características, sin producir regresiones en sistemas previos.
\end{itemize}
