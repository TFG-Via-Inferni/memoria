\begin{abstract}
En este capítulo se describe el marco de trabajo empleado para el desarrollo de Via Inferni.
Se presenta la estructura organizativa del equipo de dos integrantes sin roles diferenciados, 
donde ambos asumen conjuntamente todas las responsabilidades del proyecto (programación, 
diseño, arte y pruebas). Se explica la metodología ágil adaptada al equipo, basada en Kanban 
y una Definición de Hecho (DoD) consensuada, junto con un flujo de trabajo en control de 
versiones basado en Trunk-Based Development simplificado con ramas de integración de corta 
duración. Este enfoque prioriza el desarrollo conjunto del núcleo del juego para garantizar 
conocimiento compartido, seguido de la implementación paralela de contenido adicional una vez 
establecida una base arquitectónica sólida.
\end{abstract}

\section{Estructura organizacional del proyecto}

El proyecto ha sido desarrollado por un equipo de dos integrantes que asumen conjuntamente 
todos los roles necesarios en un proyecto de videojuego. A diferencia de estructuras 
tradicionales donde se asignan roles específicos (programador, diseñador, artista, tester), 
ambos integrantes participan equitativamente en todas las facetas del desarrollo: 
programación de sistemas, diseño de mecánicas, implementación de elementos visuales, 
pruebas y ajustes de \textit{gameplay}.

Este modelo de responsabilidad compartida sin roles diferenciados busca garantizar 
que ambos integrantes posean un conocimiento profundo y total de la arquitectura del 
juego, evitando silos de información y facilitando la toma de decisiones conjunta. 
Dentro de este marco de trabajo colaborativo, la distribución de tareas se organiza 
de la siguiente manera:

\begin{itemize}
    \item \textbf{Desarrollo conjunto del núcleo (Core):} Las funcionalidades base 
    que sustentan el juego (generación procedural de niveles, dualidad de personajes, 
    sistema de combate, enemigos e ítems básicos) han sido implementadas por partes iguales. 
    Ambos integrantes trabajan sobre los mismos scripts y sistemas fundamentales, 
    asegurando que la arquitectura base sea robusta y comprendida por todo el equipo.
    
    \item \textbf{Implementación de contenido adicional:} Una vez asentadas las bases 
    de las funcionalidades core, la implementación de añadidos se realiza de forma 
    independiente para agilizar la producción. Elementos como nuevos enemigos, 
    ítems específicos, variaciones de niveles o mejoras visuales se asignan individualmente, 
    siempre respetando la arquitectura común desarrollada en la fase conjunta.
    
    \item \textbf{Validación cruzada:} Aunque la adición de contenido sea individual, 
    se mantiene una comunicación constante. Ninguna funcionalidad se considera cerrada hasta 
    que es revisada y aprobada por el otro integrante, garantizando la coherencia.
\end{itemize}

\section{Metodología de desarrollo}

Para el desarrollo de Via Inferni se ha optado por una metodología ágil híbrida, 
adaptada a un equipo reducido de dos personas. Las prioridades han sido la flexibilidad, 
el conocimiento compartido y la entrega continua de funcionalidades verificables.

\subsection{Gestión de tareas}

La gestión del trabajo se realiza mediante GitHub Projects, utilizando un tablero 
Kanban digital con cuatro estados claramente diferenciados: \textbf{To-do}, 
\textbf{In progress}, \textbf{In review} y \textbf{Done}. Esta herramienta proporciona 
visibilidad en tiempo real del estado del proyecto y permite vincular automáticamente 
los commits y ramas del repositorio con las tareas correspondientes, garantizando una 
trazabilidad completa del desarrollo.

\subsubsection{Planificación semanal}

Cada domingo se celebra una reunión obligatoria de planificación donde el equipo:
\begin{itemize}
    \item Revisa el trabajo completado durante la semana anterior, analizando qué 
    funcionalidades han sido implementadas y validadas.
    \item Crea las tareas para la semana siguiente, priorizándolas según las necesidades 
    del proyecto y los hitos establecidos.
    \item Distribuye el trabajo entre ambos integrantes, respetando el principio de 
    desarrollo conjunto para funcionalidades core e independiente para contenido adicional.
\end{itemize}

\subsubsection{Flujo de estados de las tareas}

El ciclo de vida de una tarea sigue este flujo:

\begin{enumerate}
    \item \textbf{To-do:} Tareas planificadas pero no iniciadas, priorizadas según 
    su importancia y dependencias.
    \item \textbf{In progress:} Cuando un integrante comienza a trabajar en una tarea, 
    la mueve a este estado y crea su rama local correspondiente.
    \item \textbf{In review:} Una vez la implementación cumple con los criterios técnicos 
    de la Definición de Hecho (DoD) (código funcional, sin errores, integrado en el juego), 
    la tarea pasa a revisión.
    \item \textbf{Done:} La tarea solo se marca como completada tras una reunión de 
    validación conjunta donde ambos integrantes verifican que cumple los criterios de 
    aceptación. Estas reuniones pueden realizarse varias veces por semana según la 
    disponibilidad del equipo y la cantidad de tareas pendientes de revisión.
\end{enumerate}

\subsubsection{Definición de Hecho (DoD)}

Para evitar ambigüedades, se establecen criterios claros que una tarea debe cumplir 
antes de considerarse terminada:

\begin{itemize}
    \item \textbf{Criterios técnicos:} La implementación debe ser funcional dentro 
    del juego, no generar errores en la consola de Unity, y no romper sistemas existentes.
    \item \textbf{Validación conjunta:} Ambos integrantes deben probar la funcionalidad 
    en una sesión de revisión, verificando que el comportamiento es el esperado y que 
    se integra correctamente con el resto del juego.
    \item \textbf{Documentación del código:} Los scripts deben incluir comentarios 
    explicativos en secciones complejas para facilitar el mantenimiento futuro.
    \item \textbf{Trazabilidad en el repositorio:} Los commits relacionados deben estar 
    vinculados a la tarea en GitHub Projects mediante referencias en los mensajes de commit.
\end{itemize}

Esta metodología de gestión garantiza que ninguna funcionalidad se considere completa 
sin el consenso del equipo, manteniendo altos estándares de calidad y conocimiento compartido.

\subsection{Sincronización y comunicación}

Debido a la naturaleza colaborativa del proyecto, la comunicación es constante:
\begin{itemize}
    \item \textbf{Sincronización Continua:} Más allá de reuniones formales diarias, 
    se mantiene una comunicación fluida durante las sesiones de desarrollo para resolver 
    cualquier tipo de duda o conflicto al instante.
    \item \textbf{Hitos de Desarrollo:} El proyecto se estructura en torno a la consecución 
    de hitos funcionales incrementales. Estos hitos son especialmente críticos durante 
    la fase de desarrollo del núcleo (ej. "Generación procedural", "Sistema de combate 
    funcional", "Dualidad de personajes"), donde se prioriza la estabilidad 
    arquitectónica antes de añadir complejidad. Una vez establecida la base sólida, 
    los hitos posteriores se orientan a la expansión de contenido y refinamiento de 
    mecánicas existentes (ej. "Pool de 10 enemigos únicos", "Sistema de objetos interactivos").
\end{itemize}

\section{Herramientas de desarrollo}

Para facilitar la colaboración, el seguimiento del proyecto y el desarrollo del juego, 
se utilizan las siguientes herramientas:

\begin{itemize}
    \item \textbf{Unity 6.3 LTS:} Motor gráfico y de juego empleado para el desarrollo 
    de Via Inferni. Se eligió esta versión LTS (Long Term Support) para garantizar 
    estabilidad y soporte a largo plazo durante el desarrollo del proyecto.
    
    \item \textbf{Visual Studio Code:} Editor de texto utilizado para la edición de código 
    y configuración. Aunque técnicamente es un editor de texto y no un IDE completo, 
    se prefiere a Visual Studio por su ligereza y familiaridad dentro del equipo.
    
    \item \textbf{GitHub:} Plataforma de alojamiento del repositorio Git y centro 
    de colaboración del proyecto. Alberga el control de versiones, la gestión de 
    ramas y la integración continua.
    
    \item \textbf{GitHub Projects:} Herramienta de gestión de tareas integrada en GitHub, 
    utilizada para mantener el tablero Kanban del proyecto con los estados 
    (To-do, In progress, In review, Done).
    
    \item \textbf{Git:} Sistema de control de versiones distribuido empleado para 
    gestionar el código fuente del proyecto localmente y sincronizarlo con el repositorio 
    remoto.
    
    \item \textbf{Clockify:} Aplicación de seguimiento de tiempo utilizada para registrar 
    las horas de trabajo dedicadas a cada tarea, permitiendo un análisis posterior del 
    esfuerzo invertido en diferentes aspectos del proyecto.
    
    \item \textbf{Discord:} Plataforma de comunicación sincrónica utilizada para 
    videoconferencias, reuniones de planificación y sincronización continua durante 
    las sesiones de desarrollo.
    
    \item \textbf{WhatsApp:} Aplicación de mensajería instantánea utilizada para 
    comunicaciones rápidas y asincrónicas fuera de las sesiones de desarrollo formales.
\end{itemize}

\section{Control de versiones y flujo de trabajo}

La gestión del código fuente es vital para evitar conflictos en un motor 
basado en componentes como Unity. Se ha adoptado un enfoque de Trunk-Based Development 
simplificado, que prioriza la integración continua en ramas de vida corta y minimiza 
la complejidad del repositorio remoto.

\subsection{Flujo de trabajo con Git}

El repositorio se estructura en torno a dos ramas persistentes:

\begin{itemize}
    \item \textbf{Rama Main:} Contiene únicamente las versiones estables y 
    validadas del proyecto. Cada domingo, durante la reunión semanal, si el equipo 
    acuerda que todo el trabajo en \texttt{develop} se ha completado correctamente, 
    se realiza un merge hacia \texttt{main}, generando así una nueva versión estable.
    
    \item \textbf{Rama Develop:} Es la rama de integración principal donde convive 
    el código en desarrollo activo. Ambos integrantes sincronizan su trabajo 
    constantemente contra esta rama.
\end{itemize}

El desarrollo de nuevas funcionalidades sigue un flujo sistemático basado en 
ramas de corta duración que no se publican en el repositorio remoto. 
A continuación se detalla el proceso completo:

\subsubsection{Proceso de desarrollo de una funcionalidad}

\begin{enumerate}
    \item \textbf{Sincronización inicial:} Antes de comenzar cualquier trabajo, 
    se actualiza la rama \texttt{develop} local con los últimos cambios del repositorio remoto:
    \begin{verbatim}
    git checkout develop
    git pull
    \end{verbatim}
    
    \item \textbf{Creación de rama de funcionalidad:} Se crea una rama local 
    siguiendo la convención de nombres \texttt{feat/nombre-funcionalidad}:
    \begin{verbatim}
    git checkout -b feat/nombre-funcionalidad
    \end{verbatim}
    
    \item \textbf{Desarrollo y commits atómicos:} El trabajo se realiza mediante 
    commits atómicos (pequeños y con propósito único) siguiendo la especificación 
    Conventional Commits\footnote{https://www.conventionalcommits.org/}:
    \begin{verbatim}
    git add .
    git commit -m "feat: implementar funcionalidad"
    \end{verbatim}
    
    Los prefijos más utilizados son: \texttt{feat:} (nueva funcionalidad), 
    \texttt{fix:} (corrección de errores), \texttt{refactor:} (reestructuración 
    sin cambio funcional), \texttt{docs:} (documentación).
    
    \item \textbf{Sincronización frecuente con develop:} Periódicamente, 
    especialmente antes de finalizar una funcionalidad, se sincronizan los 
    cambios de \texttt{develop} mediante rebase para evitar conflictos posteriores:
    \begin{verbatim}
    git fetch origin
    git rebase origin/develop
    \end{verbatim}
    
    \item \textbf{Preparación para integración:} Una vez la funcionalidad cumple 
    con la Definición de Hecho (DoD), se realiza una sincronización final:
    \begin{verbatim}
    git checkout develop
    git pull
    git checkout feat/nombre-funcionalidad
    git rebase develop
    \end{verbatim}
    
    \item \textbf{Integración fast-forward:} Se integra la funcionalidad en 
    \texttt{develop} mediante un merge fast-forward, manteniendo un historial lineal:
    \begin{verbatim}
    git checkout develop
    git merge --ff-only feat/nombre-funcionalidad
    \end{verbatim}
    
    \item \textbf{Publicación:} Los cambios se suben al repositorio remoto:
    \begin{verbatim}
    git push origin develop
    \end{verbatim}
    
    \item \textbf{Limpieza:} Finalmente, se elimina la rama de funcionalidad local, 
    ya que su ciclo de vida ha terminado:
    \begin{verbatim}
    git branch -d feat/nombre-funcionalidad
    \end{verbatim}
\end{enumerate}

Este enfoque garantiza un historial limpio y lineal en el repositorio remoto, 
permite el trabajo en paralelo sin interferencias, y asegura que todo el código 
en \texttt{develop} ha sido integrado y probado correctamente antes de llegar a producción.
