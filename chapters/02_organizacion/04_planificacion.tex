\begin{abstract}
Este capítulo detalla la organización temporal y la gestión de recursos del proyecto \textit{Via Inferni}. La planificación se estructura en tres fases principales que cubren desde la concepción inicial hasta el cierre definitivo de la memoria. Además, se presentan los diferentes planes de gestión basados en las mejores prácticas del estándar PMBOK, adaptados a la escala y necesidades de un Trabajo de Fin de Grado desarrollado por dos alumnos de ingeniería. El objetivo es garantizar una ejecución ordenada que cumpla con los plazos académicos y los requisitos de calidad técnica.
\end{abstract}

\section{Estructura de Fases del Proyecto}
El desarrollo se ha dividido en tres grandes etapas que permiten una transición lógica entre la teoría, la implementación y la documentación final:

\subsection{Fase 1: Planificación y Gestión}
Esta etapa comprende el establecimiento de los cimientos del proyecto. Se centra en el análisis de requisitos, la definición de objetivos técnicos y la configuración de toda la infraestructura necesaria, como el repositorio de código, el sistema de gestión de tareas y la elección del \textit{stack} tecnológico. Es aquí donde se desarrolla el modelo general de FDD que servirá de guía para el resto del trabajo.

\subsection{Fase 2: Diseño e Implementación}
Representa el grueso del trabajo técnico y se desarrolla de forma puramente iterativa. En esta fase, se seleccionan elementos de la lista de funcionalidades para ser diseñados, implementados y validados. La duración total de esta etapa se subdivide en varias iteraciones de aproximadamente tres semanas cada una. Al final de cada ciclo, el equipo obtiene un incremento funcional del videojuego que se añade a la rama de desarrollo.

\subsection{Fase 3: Cierre del Proyecto}
La fase final se dedica a la consolidación de la memoria técnica, la realización de las pruebas finales de integración y la preparación del material necesario para la defensa del trabajo. En esta etapa se asegura que toda la documentación técnica refleje fielmente la implementación final y se corrigen posibles inconsistencias detectadas durante el desarrollo.

\section{Planes de Gestión del Proyecto}
Para dotar al proyecto de un rigor profesional, se han aplicado conceptos extraídos del PMBOK, seleccionando los procesos que aportan mayor valor a un equipo reducido de ingeniería de software.

\subsection{Gestión del Alcance}
El alcance se gestiona mediante el \textit{Feature List} definido en la metodología FDD. Para evitar el fenómeno de la corrupción del alcance, cualquier propuesta de nueva funcionalidad durante el desarrollo debe ser sometida a un análisis de viabilidad temporal. Si la tarea compromete el cronograma de la Fase 2, se clasifica como una mejora para futuros desarrollos.

\subsection{Gestión de Tiempos y Calendario}
Se estima una dedicación de 300 horas por miembro del equipo. El calendario se rige por hitos de control que coinciden con el final de cada fase y cada iteración. La visualización de este cronograma permite identificar desviaciones de forma temprana y ajustar la carga de trabajo de las siguientes iteraciones para cumplir con la fecha de entrega final.

\subsection{Gestión de la Comunicación y RRHH}
Al ser un equipo de dos personas, la comunicación es constante y bidireccional. No obstante, se establecen canales formales para evitar la pérdida de información:
\begin{itemize}
    \item Se utiliza Discord para la coordinación técnica diaria y la resolución inmediata de bloqueos.
    \item Se realizan reuniones semanales de sincronización para actualizar el tablero Kanban y revisar la planificación de la iteración en curso.
    \item El uso de GitHub para las revisiones de código asegura que ambos miembros tengan una visión global de todo el software desarrollado.
\end{itemize}

\subsection{Gestión de Riesgos}
Se han identificado y priorizado los riesgos que podrían impactar en el éxito del proyecto:
\begin{itemize}
    \item \textbf{Riesgos Técnicos:} Problemas de rendimiento con el algoritmo de generación procedural. Se mitiga mediante la creación de prototipos tempranos o \textit{spikes} de investigación.
    \item \textbf{Riesgos de Plazos:} Retrasos debidos a la complejidad de la integración de sistemas. Se mitiga manteniendo un margen de seguridad en la fase de cierre y priorizando las funcionalidades críticas del núcleo jugable.
\end{itemize}
