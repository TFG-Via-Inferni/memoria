\chapter{Plan de Dirección del Proyecto}

\begin{quotation}[General and President]{Dwight D. Eisenhower (1890--1969)}
Plans are useless, but planning is indispensable.
\end{quotation}

\begin{abstract}
En este capítulo se define el Plan de Dirección del Proyecto de \textit{Via Inferni},
tomando como referencia el enfoque propuesto por PMBOK y adaptándolo a la escala real de
un Trabajo de Fin de Grado desarrollado por dos personas. El objetivo no es reproducir una
gestión documental pesada, sino establecer un marco integrado que explique cómo se
dirige, planifica, ejecuta, supervisa y cierra el proyecto \cite{pmbok2021}.
\end{abstract}

\section{Propósito del Plan de Dirección}
El Plan de Dirección del Proyecto actúa como documento integrador de la planificación. En
PMBOK \cite{pmbok2021}, este plan coordina los planes subsidiarios de gestión, las líneas base del proyecto
y los mecanismos de control que permiten comparar lo planificado con lo realmente
ejecutado.

En el caso de \textit{Via Inferni}, este plan se adapta a un contexto académico y a un
equipo reducido. Por ello, se mantienen los elementos que aportan control real sobre el
proyecto y se simplifican aquellos que, por la naturaleza del TFG, introducirían más carga
administrativa que valor. La planificación se orienta a asegurar tres aspectos:

\begin{itemize}
    \item que el alcance del videojuego sea viable dentro del tiempo disponible;
    \item que el desarrollo avance mediante incrementos jugables y verificables;
    \item que las decisiones técnicas y de gestión queden documentadas de forma trazable.
\end{itemize}

\section{Enfoque de Dirección basado en PMBOK}
El proyecto toma como referencia las áreas de conocimiento de PMBOK, pero las aplica de
forma proporcional. En lugar de desarrollar todos los planes con el mismo nivel de detalle,
se priorizan las áreas que tienen impacto directo sobre el éxito del TFG: integración,
alcance, cronograma, costes, calidad, comunicaciones y riesgos.

Esta adaptación permite mantener una visión de proyecto completa sin perder agilidad. La
planificación sirve como marco de decisión, mientras que la ejecución técnica se documenta
de forma incremental y vinculada a entregables funcionales.

\subsection{Gestión de la integración}
La gestión de la integración se materializa en este propio Plan de Dirección. Su función es
unificar el enfoque metodológico, los planes subsidiarios, las líneas base y los criterios
de seguimiento. Cualquier decisión relevante sobre alcance, calendario, calidad o riesgos
debe ser coherente con este marco.

El control integrado del proyecto se apoya en reuniones de planificación y cierre de
iteración, revisión del backlog, seguimiento del tablero Kanban y contraste continuo entre
lo implementado y lo documentado en la memoria.

\subsection{Planes subsidiarios incluidos}
Los planes de gestión desarrollados para este proyecto son los siguientes:

\begin{itemize}
    \item \textbf{Plan de gestión del alcance:} define cómo se delimita el producto y cómo
    se controlan nuevas funcionalidades o cambios.
    \item \textbf{Plan de gestión del cronograma:} organiza el trabajo en fases e
    iteraciones, con hitos y una distribución temporal planificada.
    \item \textbf{Plan de gestión de comunicaciones:} establece los canales y rutinas de
    coordinación del equipo.
    \item \textbf{Plan de gestión de riesgos:} identifica las incertidumbres principales y
    las respuestas previstas.
    \item \textbf{Plan de gestión de la calidad:} fija los criterios que debe cumplir una
    funcionalidad para considerarse terminada.
    \item \textbf{Plan de gestión de costes:} estima el valor económico del esfuerzo y los
    recursos utilizados.
\end{itemize}

Estos planes se desarrollan en detalle en el capítulo siguiente, donde se concreta su
aplicación al proyecto.

\subsection{Líneas base del proyecto}
El Plan de Dirección queda apoyado por tres líneas base principales:

\begin{itemize}
    \item \textbf{Línea base del alcance:} formada por el modelo general del producto y la
    lista de funcionalidades priorizadas mediante FDD.
    \item \textbf{Línea base del cronograma:} formada por las fases del proyecto, las ocho
    iteraciones de desarrollo y los hitos de cierre.
    \item \textbf{Línea base de costes:} formada por la estimación de horas, coste de
    personal, amortización de equipamiento y costes indirectos.
\end{itemize}

Estas líneas base permiten comparar la planificación inicial con la evolución real del
proyecto y justificar las desviaciones que aparezcan durante la ejecución.

\section{Ciclo de vida del proyecto}
El ciclo de vida del proyecto se organiza en tres fases principales:

\begin{itemize}
    \item \textbf{Planificación:} definición de objetivos, alcance, metodología, herramientas,
    repositorio, backlog inicial y planes de gestión.
    \item \textbf{Ejecución, seguimiento y control:} desarrollo iterativo del videojuego,
    validación de funcionalidades, revisión de riesgos y actualización de la documentación.
    \item \textbf{Cierre:} consolidación del producto, revisión final de la memoria,
    preparación de manuales y formalización de conclusiones.
\end{itemize}

Esta estructura combina una planificación inicial suficientemente sólida con una ejecución
iterativa que permite ajustar el trabajo conforme se obtiene conocimiento técnico del
videojuego.

\section{Metodología de Desarrollo}
Para la construcción de \textit{Via Inferni} se ha seleccionado la metodología
\textit{Feature Driven Development} (FDD) \cite{palmer2002fdd}. La razón principal es que encaja especialmente
bien en proyectos donde la funcionalidad debe crecer por incrementos pequeños,
comprobables y con un diseño técnico suficientemente claro antes de implementar cada
cambio.

FDD permite trabajar con una visión global del producto, pero sin perder granularidad en
la ejecución. En lugar de abordar el desarrollo como un bloque único, el proyecto se
descompone en funcionalidades manejables que pueden diseñarse, implementarse y validarse
por separado.

El proceso se articula a través de cinco pasos fundamentales que guían el proyecto desde
su concepción hasta la incorporación de cada funcionalidad, tal y como se ilustra en la
Cuadro \ref{fig:fdd_flow}.

\section{Gestión de la configuración y el código}
La gestión del repositorio es crítica para asegurar la integridad del proyecto. Para ello
se ha definido una estrategia híbrida que combina la organización estructural de
\textit{GitFlow} \cite{driessen2010gitflow} con la integración frecuente propia de \textit{Trunk-Based Development} \cite{trunkbasedIntro}.

La rama \texttt{main} representa el estado estable del proyecto, mientras que
\texttt{develop} actúa como punto de integración del trabajo validado. Las ramas de
funcionalidad son efímeras y se crean para resolver tareas concretas del \textit{backlog},
reduciendo así el coste de integración y el riesgo de divergencias largas.

Como criterio de trazabilidad se ha adoptado el estándar \textit{Conventional Commits} \cite{conventionalCommits10},
que facilita identificar la intención de cada cambio y relacionarlo con funcionalidades,
correcciones o ajustes de documentación.

\section{Sistema de control de cambios}
Los cambios relevantes del proyecto se gestionan mediante un criterio proporcional a su
impacto. Las correcciones menores o ajustes técnicos que no afectan a la planificación se
resuelven dentro de la iteración correspondiente. En cambio, una modificación que altere el
alcance, desplace hitos o introduzca riesgos nuevos debe discutirse entre los dos miembros
del equipo y registrarse en el backlog o en la documentación de la iteración.

Este sistema evita que el proyecto incorpore trabajo no planificado sin evaluar su impacto.
La decisión de aceptar, dividir, posponer o descartar una funcionalidad se toma comparando
su valor para el núcleo jugable con el coste temporal y técnico que supone.

\section{Herramientas y tecnologías}
El desarrollo del proyecto se apoya en un conjunto de herramientas técnicas y de gestión
que permiten coordinar el trabajo y mantener control sobre el avance.

\subsection{Criterios de selección tecnológica}
Antes de fijar el entorno de trabajo se valoraron distintas alternativas para el desarrollo
del proyecto. La elección no se basó únicamente en la popularidad de las herramientas, sino
en su adecuación al tipo de videojuego planteado, a la escala real del TFG y al nivel de
madurez técnica del equipo.

En el caso del motor de desarrollo, se consideró especialmente la conveniencia de trabajar
con una tecnología capaz de soportar con solvencia un videojuego 2D de tipo
\textit{roguelike}, con generación procedural, gestión de escenas, interfaz, combate y una
iteración rápida sobre sistemas jugables. Tras revisar opciones habituales en este ámbito y
comparar sus características de forma general, se concluyó que Unity ofrecía el equilibrio
más adecuado entre capacidad técnica, documentación disponible, curva de aprendizaje
asumible y velocidad de prototipado \cite{unityLearnGettingStarted}.

La decisión de utilizar GitHub como base del control de versiones y de la trazabilidad del
trabajo también fue deliberada. Aunque existían alternativas más específicas para proyectos
de Unity, se optó por Git y GitHub porque era el flujo de trabajo ya consolidado en la
formación previa del equipo y porque ofrecía una trazabilidad más completa para los
objetivos de la memoria: historial de cambios, revisión de código, gestión de incidencias,
ramas de trabajo y vinculación directa entre desarrollo y documentación del proceso \cite{githubIssuesProjects}.

Por último, la memoria se mantuvo en LaTeX \cite{latexProject} porque encajaba mejor con un documento extenso,
técnico y modular. Su uso permitió separar capítulos, controlar referencias cruzadas,
figuras, tablas y bibliografía con un nivel de rigor coherente con el tratamiento dado al
código fuente y al resto de artefactos del proyecto.

Para el registro de dedicación temporal se incorporó además Clockify \cite{clockifyOfficial}. La elección fue
práctica: se trata de una herramienta simple, suficientemente ligera para un equipo
reducido y ya conocida por ambos integrantes del proyecto.

\begin{itemize}
    \item Unity actúa como motor gráfico para el desarrollo del videojuego \cite{unityLearnGettingStarted}.
    \item Git y GitHub se utilizan para el control de versiones, revisión de cambios y seguimiento de tareas \cite{githubIssuesProjects}.
    \item GitHub Projects proporciona el tablero Kanban con el que se monitoriza el flujo de trabajo \cite{githubIssuesProjects}.
    \item Discord se utiliza como canal principal de comunicación operativa.
    \item Clockify se utiliza para registrar las horas de trabajo reales de cada integrante del equipo \cite{clockifyOfficial}.
    \item La memoria se mantiene en LaTeX, lo que permite modularizar capítulos y gestionar el documento con el mismo rigor que el código \cite{latexProject}.
\end{itemize}

\section{Criterios de seguimiento y cierre}
El seguimiento del proyecto se realiza de forma iterativa. Al cierre de cada ciclo se
revisa el estado de las funcionalidades, se comprueba el cumplimiento de la
\textit{Definition of Done}, se actualiza el backlog y se documentan desviaciones o
aprendizajes relevantes.

El cierre del proyecto se considera alcanzado cuando el producto dispone de un núcleo
jugable verificable, la memoria refleja de forma coherente la evolución del desarrollo y
los materiales finales permiten instalar, comprender y evaluar el trabajo realizado.
